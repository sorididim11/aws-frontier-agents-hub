워크플로우 오케스트레이션 (Workflow Orchestration)
1. 계획 모드 기본 설정 (Plan Node Default)
사소하지 않은 모든 작업(3단계 이상의 절차 또는 아키텍처 결정)에 대해서는 계획 모드로 진입하십시오.
상황이 잘못되면 즉시 중단하고 계획을 다시 세우십시오. 억지로 밀어붙이지 마십시오.
구축뿐만 아니라 검증 단계에서도 계획 모드를 사용하십시오.
모호함을 줄이기 위해 사전에 상세한 사양을 작성하십시오.
2. 하위 에이전트 전략 (Subagent Strategy)
메인 컨텍스트 창을 깨끗하게 유지하기 위해 하위 에이전트를 자유롭게 사용하십시오.
조사, 탐색 및 병렬 분석은 하위 에이전트에게 맡기십시오.
복잡한 문제의 경우, 하위 에이전트를 통해 더 많은 연산 자원을 투입하십시오.
집중적인 실행을 위해 하위 에이전트당 하나의 작업을 할당하십시오.
3. 자기 개선 루프 (Self-Improvement Loop)
사용자로부터 수정을 받은 후에는 반드시 해당 패턴을 tasks/lessons.md에 업데이트하십시오.
동일한 실수를 방지하기 위한 스스로의 규칙을 작성하십시오.
실수율이 떨어질 때까지 이러한 교훈들을 가차 없이 반복 적용하십시오.
세션 시작 시 관련 프로젝트에 대한 교훈들을 검토하십시오.
4. 완료 전 검증 (Verification Before Done)
작동하는 것을 증명하기 전까지는 작업을 완료로 표시하지 마십시오.
관련이 있는 경우 메인 동작과 변경 사항 간의 차이점을 비교하십시오.
스스로에게 물어보십시오: "스태프 엔지니어가 이것을 승인할 것인가?"
테스트를 실행하고, 로그를 확인하며, 정확성을 입증하십시오.
5. 우아함 요구 (균형 유지) (Demand Elegance (Balanced))
사소하지 않은 변경의 경우: 잠시 멈추고 "더 우아한 방법이 있을까?"라고 자문해 보십시오.
해결책이 임시방편처럼 느껴진다면: "지금 알고 있는 모든 것을 바탕으로, 우아한 솔루션을 구현하라"고 생각하십시오.
간단하고 명확한 수정의 경우에는 이를 건너뛰십시오. 과잉 엔지니어링을 하지 마십시오.
결과물을 제시하기 전에 스스로의 작업에 이의를 제기해 보십시오.
6. 자율적 버그 수정 (Autonomous Bug Fixing)
버그 보고를 받으면 그냥 수정하십시오. 일일이 도움을 요청하지 마십시오.
로그, 오류, 실패한 테스트를 지적한 다음 해결하십시오.
사용자로부터 어떠한 컨텍스트 전환도 요구하지 마십시오.
지시를 받지 않고도 실패한 CI 테스트를 수정하십시오.
작업 관리 (Task Management)
계획 우선: 체크 가능한 항목이 포함된 계획을 .kiro/specs/{feature}/tasks.md에 작성하십시오.
계획 검증: 구현을 시작하기 전에 확인을 받으십시오.
진행 상황 추적: 진행하면서 항목을 완료로 표시하십시오.
변경 사항 설명: 각 단계에서 상위 수준의 요약을 제공하십시오.
결과 문서화: .kiro/specs/{feature}/tasks.md에 검토 섹션을 추가하십시오.
교훈 기록: 수정 사항이 발생한 후 tasks/lessons.md를 업데이트하십시오.
핵심 원칙 (Core Principles)
단순함 우선: 모든 변경 사항을 가능한 한 단순하게 만드십시오. 코드에 미치는 영향을 최소화하십시오.
게으름 금지: 근본 원인을 찾으십시오. 임시 처방은 없습니다. 시니어 개발자 수준의 기준을 유지하십시오.
최소한의 영향: 변경 사항은 필요한 부분만 건드려야 합니다. 버그 도입을 방지하십시오.

배포 규칙 (Deployment Rules)
- Dashboard 배포: 반드시 `services/dashboard/deploy.sh`를 사용하십시오. 수동 docker build/push/kubectl apply 금지.
  - deploy.sh가 빌드 → push → ECR 이미지 존재 확인 → kustomize 배포 → rollout 대기 → port-forward를 순차 실행합니다.
  - ECR에 이미지가 없으면 배포를 중단하는 가드레일이 포함되어 있습니다.
- DockerCoins 서비스(hasher, rng, worker, webui): git push → GitHub Actions 자동 빌드/배포.
  - 수동 kubectl set image, kubectl apply 금지. 항상 GitHub Actions를 통해 배포하십시오.
  - kustomize로 이미지 태그를 관리합니다. latest 태그 사용 금지.
- 이미지 태그: 항상 타임스탬프 또는 git sha 태그를 사용하십시오. latest 태그는 절대 사용하지 마십시오.
- port-forward: `scripts/port-forward.sh`를 사용하십시오. 자동 재연결이 포함되어 있습니다.
