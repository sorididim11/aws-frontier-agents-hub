## 용어집 (Glossary)

이 문서에서 반복적으로 사용되는 용어를 먼저 정리한다.

| 용어 | 뜻 | 비유 |
|------|-----|------|
| **코어 루프 (Core Loop)** | 사용자 입력 → LLM 호출 → 도구 실행 → 응답 → 저장의 반복 사이클. 에이전트의 심장 박동. | 공장의 생산 라인. 원재료 투입부터 완성품 출하까지의 한 사이클. |
| **스킬 (Skill)** | 에이전트가 특정 상황에서 참조하는 구조화된 지식. SKILL.md 파일로 존재. 코드가 아니라 마크다운 텍스트. | 요리사의 레시피 카드. "이 상황에서는 이렇게 하라"는 지침서. |
| **프롬프트 (Prompt)** | LLM에게 보내는 전체 입력 텍스트. 시스템 규칙 + 컨텍스트 + 사용자 메시지로 구성. | 직원에게 주는 업무 지시서. 회사 규칙 + 오늘 할 일 + 방금 받은 요청. |
| **프로바이더 (Provider)** | LLM 서비스 제공자. OpenAI, Anthropic, Bedrock 등. | 택배 회사. 어떤 회사를 쓰든 "물건을 보내고 받는다"는 동작은 같음. |
| **토큰 (Token)** | LLM이 텍스트를 처리하는 단위. 대략 한국어 1글자 ≈ 1~2토큰. 토큰 = 비용. | 택시 미터기. 텍스트가 길수록 비용 증가. |
| **도구 (Tool)** | 에이전트가 외부와 상호작용하는 수단. kubectl 실행, 파일 읽기 등. | 요리사의 칼, 냄비, 오븐. 각각 다른 작업에 쓰이는 도구. |
| **서브에이전트 (Sub-agent)** | 메인 에이전트가 특정 작업을 위임하는 하위 에이전트. 독립적으로 동작하고 결과만 반환. | 주방장이 디저트 담당 셰프에게 "케이크 만들어줘"하고 맡기는 것. |
| **데몬 스레드 (Daemon Thread)** | 메인 프로세스가 종료되면 자동으로 같이 종료되는 백그라운드 스레드. 사용자를 차단하지 않고 뒤에서 동작. | 밤에 자동으로 돌아가는 로봇 청소기. 주인이 자고 있어도 알아서 청소. |
| **FTS5** | SQLite의 Full-Text Search 확장. 텍스트 전문 검색을 빠르게 해주는 엔진. | 도서관 검색 시스템. "OOM"을 치면 관련 모든 대화 기록을 찾아줌. |
| **Prefix Cache** | LLM API에서 이전에 보낸 프롬프트의 앞부분이 동일하면 재계산 없이 캐시 사용. 비용 절감. | 식당에서 매번 같은 주문을 하면 "아 그거요?"하고 바로 처리하는 것. |
| **Umbrella Skill (우산 스킬)** | 여러 좁은 스킬을 하나로 묶은 상위 카테고리 스킬. | "이탈리안 요리" 레시피북. 파스타, 피자, 리조또 개별 레시피를 하나로 합친 것. |
| **Archive (아카이브)** | 삭제가 아닌 보관. .archive/ 폴더로 이동. 검색 가능하고 복원 가능. | 서류 캐비넷. 당장 안 쓰지만 버리지 않고 보관. |
| **열린 루프 vs 닫힌 루프** | 닫힌 루프: 실행 결과가 다음 실행에 영향 없음. 열린 루프: 실행 결과가 다음 실행을 개선함. | 닫힌: 실수해도 반성 없이 반복. 열린: 실수 → 복기 → 다음엔 안 함. |
| **턴 (Turn)** | 사용자의 한 번의 입력 → 에이전트의 한 번의 응답까지의 단위. | 대화에서 "한 마디 주고받기". |
| **넛지 (Nudge)** | 부드러운 유도. "이걸 스킬로 저장할까?"라는 제안. 강제가 아닌 권유. | "우산 가져가세요~" 같은 안내 방송. |

## 1. 메타 오케스트레이터란 무엇인가?

### 비유로 이해하기

**일반 오케스트레이터** = 식당 주방장
- 주문을 받으면 → 레시피대로 요리해서 → 서빙한다
- 다음 주문도 동일한 레시피로 처리한다
- 레시피를 바꾸려면 주방장(개발자)이 직접 고쳐야 한다

**메타 오케스트레이터** = 식당 주방장 + 자기 교육 시스템
- 주문을 받으면 → 레시피대로 요리해서 → 서빙한다
- **추가로**: 서빙 후 "이 요리 잘 됐나?" 복기하고
- "다음에는 소금을 줄여야지" 같은 메모를 스스로 남기고
- 7일마다 메모장을 정리해서 **레시피 자체를 업데이트**한다
- 다음 주문에서는 개선된 레시피로 요리한다

### 한 문장 정의

> **메타 오케스트레이터**: 태스크를 수행하면서 동시에 **자기 자신의 수행 능력을 자동으로 개선**하는 시스템

### 3 레벨로 본 차이

```
┌──────────────────────────────────────────────────────────────────────┐
│ Level 0: 단순 도구                                                    │
│   "kubectl get pods 실행해줘" → 실행 → 결과 반환                        │
│   (한 번의 명령, 한 번의 응답)                                          │
├──────────────────────────────────────────────────────────────────────┤
│ Level 1: 오케스트레이터 (우리 앱의 현재 위치)                             │
│   "이 서비스에 장애를 주입하고 Agent가 진단하는지 테스트해줘"                │
│   → 시나리오 생성 → 주입 → 검증 → 복원 → 결과 저장                       │
│   (여러 단계를 조율하지만, 매번 같은 방식으로 동작)                        │
├──────────────────────────────────────────────────────────────────────┤
│ Level 2: 메타 오케스트레이터 (Hermes가 하는 것)                          │
│   Level 1과 동일하게 태스크 수행 +                                      │
│   "방금 시나리오가 실패했는데, 왜 그런지 분석해서                           │
│    다음번엔 더 나은 시나리오를 만들도록 내 스킬을 수정하자"                   │
│   → 자기 개선이 자동으로 일어남                                          │
└──────────────────────────────────────────────────────────────────────┘
```

### 왜 "메타"라고 부르는가?

"메타"는 "~에 대한"이라는 뜻이다:
- 메타데이터 = 데이터에 대한 데이터
- 메타인지 = 인지에 대한 인지 (내가 뭘 아는지를 아는 것)
- **메타 오케스트레이션** = 오케스트레이션에 대한 오케스트레이션 (내가 얼마나 잘 조율하는지를 조율하는 것)

## 2. Hermes의 전체 구조 — 큰 그림

### 한눈에 보는 아키텍처

```
사용자가 뭔가를 요청한다
        │
        ▼
┌──────────────────────────────────────────────────┐
│            코어 루프 (Core Loop)                    │
│                                                  │
│   ① 입력 수집                                     │
│   ② 프롬프트 조립 (스킬 + 메모리 + 메시지)           │
│   ③ LLM 호출                                     │
│   ④ 도구 실행 (필요시 반복)                         │
│   ⑤ 루프 감지 (무한루프 방지)                       │
│   ⑥ 응답 반환                                     │
│   ⑦ 저장 + ★자기 개선 트리거★                      │
│                                                  │
└────────────────────┬─────────────────────────────┘
                     │
         ┌───────────┼───────────┐
         ▼           ▼           ▼
┌─────────────┐ ┌──────────┐ ┌──────────────┐
│ Background  │ │  메모리   │ │   Curator    │
│ Review      │ │  (기억)   │ │  (정리)      │
│ (즉시 개선)  │ │          │ │  (7일 주기)   │
└──────┬──────┘ └────┬─────┘ └──────┬───────┘
       │             │              │
       └─────────────┴──────────────┘
                     │
                     ▼
         ┌───────────────────────┐
         │  개선된 스킬/메모리     │
         │  → 다음 요청에서 사용   │
         └───────────────────────┘
```

### 왜 이 구조가 강력한가?

일반적인 에이전트 시스템은 "실행 → 결과 → 끝"이다.
Hermes는 "실행 → 결과 → **복기** → 스킬 수정 → 다음 실행이 더 나아짐"이다.

**핵심 통찰**: 코어 루프의 마지막 단계(⑦ 저장)가 단순 로그가 아니라, **자기 개선의 시작점**이라는 것. 이것이 "닫힌 루프"와 "열린 루프"의 차이.

## 3. 코어 루프 — 단계별 상세 설명

### 왜 코어 루프를 이해해야 하는가?

코어 루프는 에이전트의 **심장**이다. 사용자와의 모든 상호작용이 이 루프를 한 번 이상 통과한다.
자기 개선 시스템도 이 루프의 결과물을 기반으로 동작하므로, 루프를 이해해야 전체가 보인다.

### Stage ①: 입력 수집 (Input)

**하는 일**: 사용자 메시지를 받고, 이번 대화에 필요한 배경 정보를 모은다.

```
사용자: "EKS 클러스터의 OOM 문제를 진단해줘"
        │
        ▼
시스템이 모으는 것:
  - 이 메시지 자체
  - 이전 대화 히스토리 (같은 세션이면)
  - 현재 활성화된 스킬 목록
  - 이 에이전트의 역할 (Orchestrator인지 Leaf인지)
```

**핵심 설계 결정**: Hermes는 CLI든 API든 웹이든 **하나의 AIAgent 클래스**로 모두 처리한다.
진입점이 다르더라도 내부 루프는 동일. 이래야 자기 개선이 일관되게 작동한다.

---

### Stage ②: 프롬프트 조립 (Prompt Construction)

**하는 일**: LLM에게 보낼 전체 프롬프트를 조립한다. 단순히 사용자 메시지만 보내는 게 아니라, **3개 계층**으로 구성된 프롬프트를 만든다.

#### 3계층 프롬프트 아키텍처

비유: 샌드위치 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│  🍞 Stable Layer (빵 — 항상 같음)                                     │
│                                                                     │
│  내용: 시스템 프롬프트, 핵심 규칙, 에이전트의 정체성                       │
│  예시: "너는 DevOps 전문 에이전트다. 다음 규칙을 따라라..."                │
│                                                                     │
│  특징: 세션 간에도 변하지 않음                                          │
│  이점: LLM API의 prefix cache 활용 → 비용 절감                         │
│  (같은 텍스트를 반복 보내도 이미 캐시되어 있으므로 과금 안 됨)              │
├─────────────────────────────────────────────────────────────────────┤
│  🥬 Context Layer (속재료 — 세션마다 다름)                               │
│                                                                     │
│  내용: 스킬 인덱스, 메모리 요약, 사용자 프로필                            │
│  예시:                                                               │
│    "사용 가능한 스킬: [k8s-diagnosis: OOM/크래시 진단법,                  │
│     aws-troubleshoot: AWS 리소스 문제 해결, ...]"                       │
│    "이 사용자는 시니어 DevOps 엔지니어, Go 전문, React 초보"               │
│                                                                     │
│  특징: 세션 시작 시 frozen snapshot으로 주입                             │
│  핵심: 스킬의 "이름+설명"만 여기에 넣음 (본문은 아직 안 넣음!)              │
│        → 스킬 200개여도 인덱스만이니 토큰 적게 소비                        │
├─────────────────────────────────────────────────────────────────────┤
│  🍖 Volatile Layer (주문에 따라 바뀌는 메인 요리)                         │
│                                                                     │
│  내용: 현재 턴의 대화 메시지, 도구 호출 결과, 호출된 스킬 본문               │
│  예시:                                                               │
│    사용자: "EKS 클러스터의 OOM 문제를 진단해줘"                           │
│    + [k8s-diagnosis 스킬 전문 2000자] (방금 호출되었으므로 여기에 주입)      │
│                                                                     │
│  특징: 매 턴마다 변경됨                                                 │
└─────────────────────────────────────────────────────────────────────┘
```

#### 왜 이렇게 나누는가?

**문제**: LLM API는 토큰당 과금이다. 매 턴마다 긴 시스템 프롬프트를 보내면 비용이 폭발.

> **보충: Prefix Cache란?**
> LLM API(Claude, GPT 등)는 프롬프트의 앞부분이 이전 호출과 동일하면 이미 계산된 결과를 재사용한다.
> 예를 들어 5000토큰의 시스템 프롬프트가 매번 동일하면, 두 번째 호출부터는 이 5000토큰에 대한 과금이 75~90% 할인된다.
> Hermes는 이걸 **의식적으로 설계**했다 — Stable을 앞에, Volatile을 뒤에 배치해서 캐시 효율을 극대화한다.
> 만약 Volatile이 앞에 있으면 매번 전체가 새로운 프롬프트가 되어 캐시가 무용지물.

**해결**: 
- Stable은 변하지 않으니 API의 prefix cache가 적용됨 → 실질 비용 0
- Context도 세션 중 변하지 않으니 cache 유지
- Volatile만 실제 비용 발생

**우리 앱과의 차이**: 우리의 `system_prompts.py`는 SCENARIO_GEN 같은 프롬프트를 매번 통째로 보낸다. 계층 분리가 없어서 캐시 최적화가 안 됨.

#### Progressive Disclosure (점진적 공개)

```
평상시 (Context Layer):
  "사용 가능한 스킬: k8s-diagnosis (OOM/크래시 진단법)"
  → 이름과 한 줄 설명만. 토큰 ~20개.

사용자가 OOM을 언급했을 때 (Volatile Layer):
  [k8s-diagnosis 스킬 전문 2000자가 여기에 주입됨]
  → 호출될 때만 전체 본문 로드. 나머지 199개 스킬은 로드 안 함.
```

이것은 **도서관 카드 목록** 같은 것이다. 책 전체를 책상 위에 올려놓는 게 아니라, 목록만 보고 있다가 필요한 책만 꺼내 읽는다.

> **보충: 왜 Progressive Disclosure가 중요한가?**
> 
> Claude의 토큰 컨텍스트 윈도우는 200K 토큰이다. 많아 보이지만:
> - 스킬 하나의 평균 크기: ~2000자 ≈ ~1500토큰
> - 스킬 200개를 전부 로드하면: 300,000토큰 (윈도우 초과!)
> - 스킬 인덱스만 로드하면: 200개 × ~20토큰 = 4,000토큰 (2%)
> 
> 따라서 인덱스만 항상 로드하고, LLM이 "이 스킬이 필요하겠다"고 판단한 순간에만
> 해당 스킬 본문을 Volatile Layer에 주입하는 것이 필수적이다.
> 
> 우리 앱의 현재 방식: arch-discover 같은 스킬을 사용할 때 **전문을 Asset API에서 가져와서
> 통째로 Agent에게 보냄**. 스킬 수가 적은 지금은 괜찮지만, 스킬이 50개 이상 되면 문제.

### Stage ③: LLM 호출 (Provider → API)

**하는 일**: 조립된 프롬프트를 LLM에게 보내고 응답을 받는다.

```
┌─────────────────────────────────────────────────────┐
│  프로바이더 추상화                                     │
│                                                     │
│  Hermes는 200+ 모델, 18+ 프로바이더를 지원한다.         │
│  코어 루프는 어떤 LLM을 쓰든 동일하게 동작.              │
│                                                     │
│  프로바이더 예시:                                     │
│  - OpenAI (GPT-4, o1, etc.)                         │
│  - Anthropic (Claude)                               │
│  - Google (Gemini)                                  │
│  - 로컬 (Ollama, vLLM)                              │
│                                                     │
│  API 모드 (프로바이더별 호출 방식):                      │
│  - chat_completions (OpenAI 호환)                    │
│  - codex_responses (o-series)                       │
│  - anthropic_messages (Claude)                      │
└─────────────────────────────────────────────────────┘
```

**우리 앱과의 비교**: 우리도 `ai_provider.py`에서 bedrock/strands/agent_space를 추상화하고 있다. 이 부분은 Hermes와 구조가 유사.

### Stage ④: 도구 실행 (Tool Dispatch)

**하는 일**: LLM이 "도구를 사용하고 싶다"고 응답하면, 해당 도구를 실행하고 결과를 다시 LLM에게 돌려준다. 이걸 LLM이 "더 이상 도구가 필요 없다"고 할 때까지 반복.

```
사용자: "hasher 서비스의 메모리 사용량 확인해줘"

┌─────────── 1회차 ───────────┐
│ LLM 응답: "kubectl top pods  │
│   실행이 필요합니다"          │
│ → 도구: kubectl_query        │
│ → 결과: "hasher-pod 256Mi"   │
└──────────────┬──────────────┘
               │
┌─────────── 2회차 ───────────┐
│ LLM 응답: "limits도 확인할게요"│
│ → 도구: kubectl_query        │
│ → 결과: "limits: 512Mi"     │
└──────────────┬──────────────┘
               │
┌─────────── 3회차 ───────────┐
│ LLM 응답: "hasher는 현재     │
│   256Mi/512Mi 사용 중.       │
│   50% 사용률이므로 안전합니다" │
│ → 도구 호출 없음 → 종료       │
└─────────────────────────────┘
```

#### 특수 도구: 서브에이전트 위임 (delegate_tool)

LLM이 "이 작업은 하위 에이전트에게 맡기자"고 판단하면:

```
Parent (Orchestrator)
  "3개 서비스의 상태를 각각 확인해야 한다"
     │
     ├──▶ Child A: "hasher 서비스 확인"  (독립 실행)
     ├──▶ Child B: "worker 서비스 확인"  (독립 실행)  
     └──▶ Child C: "webui 서비스 확인"   (독립 실행)
     
     각 Child는:
     - 자기만의 대화 컨텍스트를 가짐
     - 자기만의 터미널(shell)을 가짐
     - 위험 명령은 자동 거부됨 (rm -rf 등)
     
     제한:
     - 최대 3개 동시 실행
     - 20분 타임아웃
     - 최대 3단계 깊이 (Child가 또 Child를 만드는 것)
```

#### 특수 도구: 스킬 관리 (skill_tool)

LLM이 "이 패턴을 스킬로 저장하자"고 판단하면:

```
가능한 액션:
- create: 새 SKILL.md 파일 생성
- patch:  기존 스킬의 특정 부분만 수정
- edit:   스킬 전체 재작성
- delete: .archive/로 이동 (실제 삭제 아님)
- write_file: references/templates/scripts 하위 파일 추가
- remove_file: 하위 파일 제거
```

---

### Stage ⑤: 루프 감지 (Loop Detection)

**하는 일**: 에이전트가 무한 루프에 빠졌는지 감지하고 자동으로 중단한다.

**왜 필요한가?**

에이전트는 자율적으로 도구를 호출한다. 만약 LLM이 같은 명령을 계속 반복하면?
아무도 "멈춰!"라고 안 하면 무한히 돈다. 토큰 = 돈이므로 방치하면 비용 폭발.

#### 감지 메커니즘 (검증된 구체 수치)

**소스**: `agent/tool_guardrails.py`, `agent/iteration_budget.py`

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 1. Tool Call Guardrail Controller (도구 호출 감시)                         │
│                                                                         │
│    감지 모드 3가지:                                                       │
│                                                                         │
│    ┌──────────────────┬──────────────────────────┬──────┬──────┐        │
│    │ 감지 대상         │ 설명                      │ 경고 │ 차단 │        │
│    ├──────────────────┼──────────────────────────┼──────┼──────┤        │
│    │ 정확 반복 실패     │ 같은 도구 + 같은 인자로   │ 2회  │ 5회  │        │
│    │                  │ 실패 반복                 │      │      │        │
│    ├──────────────────┼──────────────────────────┼──────┼──────┤        │
│    │ 동일 도구 실패     │ 같은 도구(다른 인자)로    │ 3회  │ 8회  │        │
│    │                  │ 실패 반복                 │      │      │        │
│    ├──────────────────┼──────────────────────────┼──────┼──────┤        │
│    │ 진행 없는 반복     │ read_file 등 읽기 도구가  │ 2회  │ 5회  │        │
│    │                  │ 동일한 결과 반환           │      │      │        │
│    └──────────────────┴──────────────────────────┴──────┴──────┘        │
│                                                                         │
│    경고(warn): 도구 실행은 되지만 "이것은 루프처럼 보인다.                   │
│                전략을 바꿔라"는 안내가 LLM에게 주입됨                        │
│    차단(block/halt): 도구 실행 자체가 방지되고 턴 강제 종료                   │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│ 2. Iteration Budget (API 호출 횟수 제한)                                  │
│                                                                         │
│    부모 에이전트: 최대 90회 API 호출/턴                                     │
│    서브에이전트:  최대 50회 API 호출/턴                                     │
│    예산 소진 후: 1회 "grace call" (요약 응답만 허용)                         │
│    execute_code 도구는 예산 환불됨 (프로그래밍적 도구 호출은 패널티 없음)       │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│ 3. Context Compression (컨텍스트 윈도우 보호)                              │
│                                                                         │
│    컨텍스트 50% 도달 → 자동 압축                                           │
│    압축 실패 시: 최대 3회 재시도                                            │
│    2회 연속 비효과적(<10% 절약) → 압축 중단 (anti-thrashing)                 │
│    → 413 에러 시 강제 압축 + 재시도                                        │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│ 4. Empty Response Recovery (빈 응답 복구)                                 │
│                                                                         │
│    도구 실행 후 빈 응답 → "도구 결과를 처리하고 계속하라" 주입               │
│    사고(reasoning)만 있고 응답 없음 → 최대 2회 재시도                       │
│    스트림 중간에 연결 끊김 → 이미 전달된 부분을 최종 응답으로 사용             │
└─────────────────────────────────────────────────────────────────────────┘
```

> **보충: 이 수치들의 의미**
>
> - "정확 반복 실패 2회 경고, 5회 차단"이 왜 이 숫자인가?
>   → 1회는 일시적 오류일 수 있다 (네트워크 끊김 등). 2회면 "어? 이상하다"고 알려주고,
>     5회면 "확실히 문제다"고 판단하고 멈춘다.
>   → 너무 빨리 멈추면 정상 재시도까지 차단, 너무 늦게 멈추면 비용 낭비.
>
> - 부모 90회, 서브에이전트 50회: 왜 다른가?
>   → 서브에이전트는 좁은 태스크를 받는다. 50회 안에 못 끝나면 잘못된 접근.
>   → 부모는 넓은 태스크(위임, 종합)를 하므로 더 많은 여유.
>
> - Prefix cache 효과: ~75% 입력 토큰 비용 절감 (Hermes 공식 측정치)

**우리 앱과의 비교**: 우리의 `bedrock_direct.py`에는 `max_rounds=20`이 있지만, 동일 호출 반복 감지나 서브에이전트 모니터링은 없음. converse loop이 같은 도구를 무한 반복할 가능성 존재.

> **보충: 왜 루프 감지가 자기 개선에 중요한가?**
> 
> 메타 오케스트레이터는 자율성이 높다 — 스스로 도구를 호출하고, 스스로 스킬을 수정한다.
> 자율성이 높아지면 "폭주" 가능성도 높아진다:
> - 같은 kubectl 명령을 무한 반복 → 의미 없는 토큰 소비
> - Background Review가 스킬을 계속 수정/복원 반복 → 무한 ping-pong
> - 서브에이전트가 에러에 빠져서 무한 재시도
> 
> 따라서 **자율성을 높이려면 반드시 안전장치도 같이 강화**해야 한다.
> 자율성 ↑ + 안전장치 ↓ = 재앙. 둘은 한 세트.

**하는 일**: 최종 텍스트 응답을 사용자에게 반환.

(이 단계는 특별한 것 없음)

### Stage ⑦: 저장 + 자기 개선 트리거 (Persist + Turn Finalizer)

**이것이 Hermes를 "메타"로 만드는 핵심 단계이다.**

일반 에이전트: 응답 반환 → 끝. 다음 턴 기다림.
Hermes: 응답 반환 → **Background Review 트리거** → 스킬 개선 → 다음 턴이 더 나아짐.

```python
# turn_finalizer.py — 매 턴 종료 시 실행되는 코드
def finalize_turn(agent, response):
    
    # 1단계: 대화 내용을 검색 가능한 DB에 저장
    save_to_fts5_db(response)
    
    # 2단계: Background Review를 트리거할지 판단
    should_review = (
        response is not None              # 유효한 응답이 있고
        and not user_interrupted           # 사용자가 중단하지 않았고
        and "skill_manage" in agent.tools  # 스킬 수정 권한이 있고
        and agent._skill_nudge_interval > 0  # 넛지 기능이 켜져 있고
        and turns_since_last >= interval   # 충분한 턴이 지남
    )
    
    if should_review:
        # 3단계: 데몬 스레드로 Background Review 시작
        # ★ 사용자를 차단하지 않음! 비동기로 실행됨 ★
        Thread(target=background_review, daemon=True).start()
```

**비유**: 식당에서 요리를 서빙한 후(Stage ⑥), 셰프가 바로 다음 주문을 받으러 가는 동시에(사용자 차단 없음), 뒤에서 견습생이 "방금 요리 어떠셨어요?" 피드백을 조용히 분석하는 것(Background Review).

## 4. 자기 개선 시스템 — 3개의 톱니바퀴

Hermes의 자기 개선은 **거창한 AI 알고리즘이 아니다**. 실용적인 3개의 서브시스템이 서로 물려 돌아가는 구조다.

### 전체 흐름 (시간축으로 보기)

```
시간 ──────────────────────────────────────────────────────────────▶

│ 턴 1     │ 턴 2     │ 턴 3     │ ... │ 턴 N     │ (7일 후) │ 턴 N+1
│          │          │          │     │          │          │
│ 사용자   │ 사용자   │ 사용자   │     │ 사용자   │  Curator │ 사용자
│ 대화     │ 대화     │ 대화     │     │ 대화     │  실행    │ 대화
│          │          │          │     │          │          │
│          │    ↓     │          │     │    ↓     │    ↓     │
│          │ Review   │          │     │ Review   │ 스킬     │ ← 개선된
│          │ (스킬    │          │     │ (스킬    │ 통합/    │   스킬로
│          │  수정)   │          │     │  생성)   │ 정리     │   동작!
│          │          │          │     │          │          │

주기:
- Background Review: 매 N턴마다 (설정에 따라)
- Curator: 7일마다 (2시간 이상 idle 시)
- Memory: 상시 (세션 시작 시 자동 로드)
```

---

### 톱니바퀴 1: Background Review (즉각적 개선)

**비유**: 프로 운동선수가 경기 직후 비디오를 보면서 "다음엔 이 동작을 고쳐야지"하고 메모하는 것.

#### 언제 실행되나?

매 턴이 아니다. **N턴마다 한 번** (설정의 `_skill_nudge_interval`에 따라).
모든 턴에 실행하면 비용이 너무 크고, 대부분의 턴은 평범하니까.

#### 무엇을 하나?

```
Background Review가 방금 끝난 대화를 읽고 찾는 것:

┌──────────────────────────────────────────────────────────────┐
│ 1. 사용자 교정 (corrections)                                   │
│    "아 그렇게 말고 이렇게 해" → 선호도 스킬로 기록              │
│                                                              │
│ 2. 워크플로우 교정                                             │
│    "순서가 A→B→C가 아니라 B→A→C야" → 기존 스킬 patch           │
│                                                              │
│ 3. 비자명한 해결법 (non-trivial solutions)                     │
│    복잡한 문제를 5회 이상 도구 호출로 해결 → 새 스킬 create      │
│                                                              │
│ 4. Outdated 스킬 발견                                         │
│    "이 스킬대로 했는데 안 되네" → 즉시 patch                    │
└──────────────────────────────────────────────────────────────┘
```

#### 구체적 예시

```
상황: 사용자와 OOM 디버깅을 5번의 도구 호출로 해결했다.

Background Review (데몬 스레드에서):
  "이 대화에서 비자명한 패턴을 발견:
   1. kubectl describe pod → OOMKilled 확인
   2. kubectl top pod → 실시간 사용량 확인
   3. kubectl get pod -o json → limits 추출
   4. CloudWatch에서 Container Insights 메트릭 조회
   5. 원인: JVM의 -Xmx가 container limit의 90%로 설정됨
   
   → 이것을 'jvm-oom-diagnosis' 스킬로 생성하자"

결과: skills/jvm-oom-diagnosis/SKILL.md가 자동 생성됨
다음에 비슷한 문제가 오면 이 스킬이 Context Layer에 인덱싱되어 있고,
LLM이 참조하여 더 빠르게 진단할 수 있다.
```

#### 수정 우선순위 (왜 중요한가)

```
1순위: 현재 로드된 스킬 patch
  → 이번 세션에서 사용된 스킬이니 가장 관련성 높음
  
2순위: 기존 우산(umbrella) 스킬 update
  → "k8s-troubleshooting" 같은 넓은 스킬에 새 섹션 추가

3순위: 지원 파일 추가
  → references/jvm-settings.md 같은 참고 자료 추가

4순위: 새 스킬 create (최후의 수단)
  → 기존 스킬로 커버 안 될 때만
  → 이유: 스킬이 무한히 늘어나면 Curator 부담 ↑
```

#### 실행 환경 (안전하게)

```
Background Review는 위험한 짓을 할 수 없도록 설계됨:

┌─────────────────────────────────────────────────┐
│  부모 에이전트를 fork (대화 히스토리 복사)          │
│  ↓                                              │
│  auto-deny 콜백 부착                             │
│  (kubectl apply, aws create 같은 쓰기 명령 차단)   │
│  ↓                                              │
│  사용 가능 도구 제한:                             │
│  - skill_manage (스킬 CRUD만)                    │
│  - memory_tools (메모리 읽기/쓰기만)              │
│  - 그 외 모든 도구 비활성화                       │
│  ↓                                              │
│  데몬 스레드에서 실행                             │
│  (사용자는 다음 대화를 바로 이어갈 수 있음)         │
└─────────────────────────────────────────────────┘
```

---

### 톱니바퀴 2: Curator (주기적 대청소)

**비유**: 집에 책이 계속 쌓이면, 7일마다 한 번씩 서재 정리하는 사서.
- 같은 주제의 책은 하나로 합치고
- 3개월 안 본 책은 창고로 옮기고
- 빈 선반을 정리한다
- **절대 버리지 않는다** (나중에 필요할 수 있으니)

#### 왜 Curator가 필요한가?

Background Review가 스킬을 계속 만들면, 시간이 지나면서:
- 비슷한 스킬이 여러 개 생김 ("k8s-oom" + "k8s-memory-issue" + "k8s-resource-limits")
- 안 쓰는 스킬이 쌓여서 인덱스가 비대해짐
- Context Layer의 스킬 인덱스 토큰이 증가 → 비용 ↑

> **보충: "스킬 폭발" 문제**
> 
> Background Review가 열심히 일하면 한 달에 스킬이 30~50개 생길 수 있다.
> 6개월 후 = 200+ 스킬. 인덱스만 해도 4000+ 토큰.
> 게다가 유사한 스킬끼리 충돌할 수 있다:
>   - "k8s-oom-diagnosis"는 "리소스 limits을 확인하라"
>   - "k8s-resource-tuning"도 "리소스 limits을 확인하라"
>   - LLM이 둘 다 참조하면 혼란
> 
> Curator는 이 "스킬 엔트로피 증가"를 주기적으로 해소하는 역할.
> 정보이론으로 보면: Background Review = 엔트로피 증가(정보 추가),
>                  Curator = 엔트로피 감소(정보 압축/정리).
> 둘의 균형이 맞아야 시스템이 건강하게 유지된다.

Curator는 이걸 자동으로 정리한다.

#### 언제 실행되나?

```
실행 조건 (모두 AND):
  ✓ curator.enabled = true
  ✓ 마지막 실행 후 168시간(7일) 경과
  ✓ 사용자가 2시간 이상 에이전트를 쓰지 않음 (idle)
  ✓ paused = false

★ 핵심: 사용 중에는 절대 실행 안 함! idle 시에만.
  (우리 앱에서 "Flask auto-reload가 분석 스레드를 kill" 하는 것과 같은 이유)
```

#### 무엇을 하나?

**Phase 1: 자동 상태 전이 (LLM 불필요, 결정론적)**

```
규칙:
- 90일 이상 미사용 → .archive/로 이동
- 30일 이상 미사용 → "stale" 마킹 (아직 삭제는 아님)
- 이전에 stale이었는데 최근 사용됨 → reactivated

이건 LLM 호출 없이 날짜만 보고 판단. 비용 0.
```

**Phase 2: LLM 리뷰 (통합 후보가 있을 때만)**

```
LLM에게 전체 스킬 목록 + 내용 요약을 보내고 묻는다:
"이 중에서 합칠 수 있는 것들이 있나? 정리할 수 있는 것은?"

LLM이 YAML로 응답:
  consolidations:
    - from: k8s-oom-diagnosis
      into: k8s-memory-troubleshooting
      reason: "OOM 진단은 메모리 문제 해결의 하위 절차"
    - from: k8s-resource-limits-guide  
      into: k8s-memory-troubleshooting
      reason: "리소스 제한 설정도 같은 우산 아래"
  prunings:
    - name: temp-session-2026-06-03-notes
      reason: "세션 특정 메모, 일반화 불가"
```

**Phase 3: 결정 실행**

3가지 통합 방법:

```
┌──────────────────────────────────────────────────────────────────┐

#### 안전장치

```
절대 건드리지 않는 것:
  ✗ 번들 스킬 (시스템 기본 제공)
  ✗ Hub에서 설치한 스킬 (외부 출처)

## 5. 서브에이전트 시스템

### 왜 필요한가?

하나의 에이전트가 모든 것을 순차적으로 처리하면 느리다.
"3개 서비스 상태를 각각 확인해"라는 요청을 순차적으로 하면 3배 시간이 걸린다.
서브에이전트로 병렬 처리하면 빠르다.

### 역할 분리: Orchestrator vs Leaf

```
Orchestrator (지휘자):
  - 서브에이전트를 생성할 수 있음
  - 태스크를 분배하고 결과를 종합
  - "큰 그림"을 보는 역할

Leaf (실행자):
  - 서브에이전트를 만들 수 없음
  - 받은 태스크만 실행하고 결과 반환
  - 위험 명령은 자동 거부됨
  - "실무"를 하는 역할
```

### 안전 제한

```
┌───────────────────────────────────────────────────┐
│ 동시 실행: 최대 3개                                 │
│ 최대 깊이: 3 (Child → Grandchild → Great-grandchild) │
│ 타임아웃: 20분                                      │
│                                                   │
│ 모니터링:                                          │
│ - 7분 30초 아무 활동 없음 → "이거 괜찮나?" 경고      │
│ - 20분 째 같은 상태 → 강제 종료                     │
│                                                   │
│ 위험 방지:                                         │
│ - Child는 위험 명령 자동 거부                       │
│ - Parent만 최종 승인 가능                          │
└───────────────────────────────────────────────────┘
```

## 6. 스킬 포맷 — SKILL.md

### 스킬이란?

에이전트가 특정 상황에서 참조하는 **구조화된 지식**이다.
코드가 아니라 마크다운 텍스트이며, LLM의 프롬프트에 주입되어 행동을 가이드한다.

### 형식

```markdown
---
name: k8s-oom-diagnosis
description: K8s 컨테이너 OOM Kill 진단 및 해결
version: 1.2.0
platforms: [linux, macos]
metadata:
  hermes:
    tags: [kubernetes, memory, troubleshooting]
---

# K8s OOM Kill 진단

## When to Use (트리거)
- 사용자가 OOM, OOMKilled, 메모리 부족을 언급할 때
- Pod가 CrashLoopBackOff이고 이유가 메모리일 때

## Procedure (절차)
1. kubectl describe pod <name> → 종료 이유 확인
2. kubectl top pod → 현재 메모리 사용량
3. kubectl get pod -o json | jq '.spec.containers[].resources'
4. Container Insights 메트릭 확인 (지난 1시간)
5. JVM 앱이면: -Xmx가 limit의 75% 이하인지 확인

## Pitfalls (함정)
- limit과 request를 혼동하지 말 것
- init container의 메모리는 별도 계산됨
- Guaranteed QoS가 아니면 노드 메모리 압박 시 kill 가능

## Verification (확인)
- 수정 후 kubectl top pod --containers로 안정성 확인
- 10분간 OOMKilled 재발 없으면 성공
```

### 생명주기

```
[생성] Background Review가 패턴 감지 → create
   ↓
[사용] 사용자 대화에서 관련 키워드 → Context Layer에서 발견 → Volatile에 주입
   ↓
[수정] 사용 중 outdated 발견 → patch / edit
   ↓
[추적] bump_use() — 사용할 때마다 카운터 증가
   ↓
[정리] Curator가 7일마다 평가:
       └── 활발 → 유지
       └── 30일 미사용 → stale 마킹
       └── 90일 미사용 → .archive/로 이동
       └── 유사 스킬 존재 → 우산에 병합
```

## 7. 우리 앱에 적용할 수 있는 것

### 닫힌 루프 vs 열린 루프 — 핵심 문제

```
┌──── 우리 앱 (현재: 닫힌 루프) ────────────────────────────────┐
│                                                             │
│  시나리오 생성 → 실행 → 결과 DDB 저장 → (끝)                  │
│                                                             │
│  다음 시나리오 생성할 때:                                      │
│  - 이전 실패 결과를 참조하지 않음                               │
│  - 같은 실수를 반복할 가능성                                   │
│  - 스킬(arch-discover, scenario-generate)이 고정적            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌──── Hermes 패턴 적용 시 (열린 루프) ─────────────────────────┐
│                                                             │
│  시나리오 생성 → 실행 → 결과 DDB 저장                          │
│       ↓                                                     │
│  Background Review (데몬 스레드):                             │
│    "이 시나리오가 실패한 이유: verification step의             │
│     expected 패턴이 실제 kubectl 출력과 불일치"                 │
│       ↓                                                     │
│  scenario-generate 스킬 patch:                               │
│    "verification step 생성 시, 실제 출력을 먼저 샘플링한 후     │
│     expected 패턴을 구성하라" 규칙 추가                        │
│       ↓                                                     │
│  다음 시나리오 생성:                                          │
│    개선된 scenario-generate 스킬이 Context에 로드됨            │
│    → verification step 품질이 자동으로 향상됨                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 구체적 적용 포인트

| Hermes 패턴 | 우리 앱 대응 | 구현 아이디어 |
|-------------|-------------|-------------|
| Background Review | 시나리오 실행 후 | 실패 시 DDB 결과 분석 → 스킬 patch 제안 |
| bump_use() | skill_manager | `list_remote` 호출 시 사용 카운터 증가, 미사용 스킬 표시 |
| Curator | skill_manager 확장 | 7일마다 유사 스킬 병합 제안, 미사용 스킬 archive |
| 루프 감지 | converse loop | 동일 도구 3회 연속 → 경고, 5회 → 강제 중단 |
| 3계층 프롬프트 | system_prompts.py | Stable(시스템)/Context(스킬 인덱스)/Volatile(턴) 분리 |
| Progressive Disclosure | skill → prompt | 스킬 인덱스만 기본 로드, 호출 시 본문 주입 |
| FTS5 세션 검색 | DDB 이벤트 | 과거 실행 결과 full-text 검색 → 유사 시나리오 참조 |

> **보충: 우리 앱에서 "열린 루프"를 구현한다면 구체적으로 무엇이 바뀌나?**
>
> **Before (현재)**:
> 1. 사용자가 "hasher에 OOM 시나리오 생성해줘" 요청
> 2. scenario-generate 스킬이 Agent에게 전달됨
> 3. Agent가 시나리오 JSON 생성
> 4. 실행 → verification step 실패 (expected="OOMKilled" but got="ContainerStatusUnknown")
> 5. DDB에 "fail" 저장 → 끝
> 6. 다음에 같은 요청하면 → 또 같은 실수 반복
>
> **After (열린 루프)**:
> 1~5번 동일
> 6. 실행 완료 후 Background Review 트리거 (데몬 스레드)
> 7. Review가 DDB 결과 분석: "expected 패턴이 너무 좁았음.
>    OOMKilled 외에 ContainerStatusUnknown도 OOM 징후"
> 8. scenario-generate 스킬에 patch: "OOM 검증 시 expected 패턴은
>    'OOMKilled|ContainerStatusUnknown|Error' 형태로 OR 패턴 사용할 것" 추가
> 9. 다음에 같은 요청하면 → 개선된 스킬 덕분에 더 robust한 시나리오 생성
>
> **핵심**: 6~9번이 자동으로 일어남. 개발자 개입 없이.
> 이것이 "닫힌 루프"에서 "열린 루프"로의 전환.

> **보충: 이것은 "학습"인가?**
>
> 정확히는 "학습"이 아니라 **"지식 축적 + 프롬프트 개선"**이다.
> - 모델 가중치가 바뀌는 것이 아니다 (fine-tuning 아님)
> - 스킬 텍스트(= 프롬프트에 주입되는 텍스트)가 개선되는 것이다
> - LLM의 행동은 프롬프트에 의해 결정되므로, 프롬프트 개선 = 행동 개선
> 
> 비유: 사람이 "공부"하면 뇌 회로가 바뀌지만,
>       AI 에이전트가 "개선"하면 참조하는 "치트시트"가 업데이트되는 것.
>       뇌는 그대로인데 치트시트가 더 좋아지니까 시험 성적이 오르는 원리.

### 검증 노트

> 초기 조사에서 "GEPA (Genetic-Pareto Prompt Evolution)"라는 유전 알고리즘 기반 시스템이
> 존재한다고 보고되었으나, 실제 코드베이스 검증 결과 **GEPA는 존재하지 않음** (LLM hallucination).
>
> 실제 Hermes의 자기 개선은:
> - 정형화된 유전 알고리즘이 **아니라**
> - **프롬프트 지시 + 데몬 아키텍처** (Background Review + Curator) 방식
> - 더 실용적이며, 우리 앱에 적용하기에도 더 현실적인 접근
